## Current state

`@alexrebula/giselle-mui` uses only standard MUI v7 APIs to set up its theme:

```ts
import { extendTheme } from '@mui/material/styles';
```

There are zero external theme utility imports in this package.
See [`theming/nextjs.md`](./theming/nextjs.md) for the recommended setup in a new project.

### Phase C — GiselleThemeProvider component — Theming (HIGH priority)

**Goal:** Expose a `<GiselleThemeProvider>` wrapper from `giselle-mui` that:

1. Ships with the Giselle brand theme (Phase B) as the default — zero-config usage
2. Accepts a `themeOverrides` prop for consumers who want a different palette, typography, or spacing
3. Accepts a `theme` prop for consumers who want to bypass the defaults entirely and pass their own `extendTheme()` result

This is the DX goal:

```tsx
// Zero config — uses Giselle green + amber palette
<GiselleThemeProvider>
  <App />
</GiselleThemeProvider>

// Consumer overrides specific tokens — still wraps in ThemeProvider correctly
<GiselleThemeProvider themeOverrides={{ palette: { primary: { main: '#1976d2' } } }}>
  <App />
</GiselleThemeProvider>

// Fully custom — consumer owns the full theme
<GiselleThemeProvider theme={extendTheme(myThemeInput)}>
  <App />
</GiselleThemeProvider>
```

**Design principle — sensible defaults, easy to override:**

The previous plan required consumers to provide all tokens. This created too much friction for
the zero-config case. The revised design ships a real default so consumers can try the library
immediately without any theme configuration.

**What it wraps:**

```tsx
// Internal implementation shape (simplified)
import { ThemeProvider, extendTheme } from '@mui/material/styles';
import { giselleTheme } from '../utils/theme-preset';

function GiselleThemeProvider({ children, themeOverrides, theme }: Props) {
  const resolvedTheme = theme ?? extendTheme(merge(giselleTheme, themeOverrides ?? {}));
  return <ThemeProvider theme={resolvedTheme}>{children}</ThemeProvider>;
}
```

### Phase G — TimelineItemDetails — Components (Medium priority)

**Goal:** Export a `<TimelineItemDetails>` component — the universal read/edit panel for
any timeline item (phase, milestone, life event, scenario). Rendered inside `DetailsDrawer`,
it displays the full item data in a structured, Asana-inspired layout.

**Why this belongs in giselle-mui:**

Every timeline variant this library will ever export (TimelineTwoColumn, the planned
RoadmapTimeline, a future ProjectTimeline) shares the same conceptual structure: an item has
a title, a date, a description, sub-items, photos, links, and status metadata. The details
view is non-trivial to implement correctly (accessible field layout, read/edit mode toggle,
photo grid, nested sub-item list). Encoding it here means no consumer ever reimplements it.

**Layout — Asana task detail model:**

```
┌────────────────────────────────────────┐
│  [Status chip]  [Date]                 │  ← top metadata row
│  Title (h2, editable in edit mode)     │
│  Short description (subtitle)          │
│                                        │
│  ── Details ─────────────────────────  │
│  Long description / body               │
│                                        │
│  ── Sub-items ───────────────────────  │
│  ○ Sub-item 1                          │
│  ○ Sub-item 2                          │
│                                        │
│  ── Photos ──────────────────────────  │
│  [img]  [img]  [img]                   │  ← row of thumbnails
│                                        │
│  ── Links / related ─────────────────  │
│  • link 1                              │
└────────────────────────────────────────┘
```

**Two modes — read and edit:**

- **Read mode (Phase G v1):** All fields rendered as static text/chips/images. No form
  controls. This is the primary deliverable.
- **Edit mode (Phase G v2, separate milestone):** Fields become controlled MUI inputs
  (`TextField`, `DatePicker`, etc.). A `onSave(updatedItem)` callback is called on submit.
  The drawer `footer` slot shows Save / Cancel. Edit mode is **not** in scope for v1.

**Data shape — generic, not timeline-specific:**

`TimelineItemDetails` accepts a `TimelineItemDetailData` interface that maps to the common
subset of `TimelinePhase` and `Milestone` from `TimelineTwoColumn`. This avoids coupling
the details component to any one timeline type:

```ts
export interface TimelineItemDetailData {
  title: string;
  shortTitle?: string;
  date?: string;
  description?: string;
  details?: string[]; // bullet list items
  photos?: Array<{ src: string; alt: string }>;
  color?: HighlightedPaletteKey;
  done?: boolean;
  overdue?: boolean;
  links?: Array<{ label: string; href: string }>;
  /** Any extra fields the consumer wants to display. */
  extra?: Array<{ label: string; value: ReactNode }>;

```

**Integration with TimelineTwoColumn:**

`TimelineTwoColumn` will accept an `onItemClick?: (item: TimelineItemDetailData) => void`
callback. When a phase card or milestone badge is clicked (not the done-dot), the callback
fires with the normalised `TimelineItemDetailData`. The consumer opens `DetailsDrawer` and
passes the data to `TimelineItemDetails`. This keeps `TimelineTwoColumn` and `DetailsDrawer`
decoupled — the consumer owns the open/close state.

### Phase J — Headless CMS / Sitecore JSS developer experience — Oct–Nov 2026

**Goal:** Make `giselle-mui` a natural fit for developers building with Sitecore JSS or any headless CMS + Next.js stack — where content is decoupled from component structure and the layout engine, not the callsite JSX, decides what renders inside a component.

**The key insight — why CMS components are rarely prop-heavy:**

In a Sitecore JSS project, a component receives `fields` (content from the CMS API) and `rendering` (layout data). Child components are not passed as props — they are wired by the Sitecore layout service through `<Placeholder>` slots. This means:

- A component cannot accept `items={[...]}` and render them as children. The CMS renders the children.
- Every zone where CMS-managed content appears must be a named `ReactNode` slot or `children`.
- Props should be optional wherever the CMS drives the content — forcing required props breaks CMS rendering.
- `data-*` attributes are attached by the Sitecore Experience Editor for in-context editing — they must flow through to the root element without configuration.

**The four patterns this phase documents and implements:**

**1 — Slot-first API design**

`ReactNode` named slots instead of data arrays. The component owns structure and styling; the consumer (or the CMS) fills each slot with whatever is appropriate.

```tsx
// ❌ prop-heavy — the component controls content shape
<FeatureCard
  title="My Title"
  description="Some text"
  ctaLabel="Learn more"
  ctaHref="/page"
/>

// ✅ slot-first — the CMS or callsite fills each zone independently
<CompositeCard
  header={<SitecoreText field={fields.title} />}
  body={<SitecoreRichText field={fields.body} />}
  actions={<Placeholder name="card-cta" rendering={rendering} />}
/>
```

**2 — Compound component pattern**

A parent component exposes React context. Named sub-components (`CompositeCard.Header`, `CompositeCard.Body`, `CompositeCard.Footer`) consume that context without prop drilling. Each sub-component can be placed in a separate Sitecore placeholder independently — the parent does not need to know the structure in advance.

```tsx
// Parent + sub-components sharing context via React.createContext
<CompositeCard color="primary">
  <CompositeCard.Header>          {/* reads color from context */}
    <SitecoreText field={fields.title} />
  </CompositeCard.Header>
  <Placeholder name="card-body" rendering={rendering} />   {/* CMS-wired */}
  <CompositeCard.Footer>
    <Placeholder name="card-actions" rendering={rendering} />
  </CompositeCard.Footer>
</CompositeCard>
```

**3 — MUI JSDoc inheritance — write docs only for new props**

When a giselle-mui component interface extends a MUI interface (`CardProps`, `ButtonBaseProps`, etc.), TypeScript inheritance carries MUI's full JSDoc into the IDE and into Storybook autodoc — with zero authoring. Only props that `giselle-mui` adds need documentation.

```ts
// MUI's JSDoc for elevation, variant, sx, aria-*, ref, etc. appears automatically.
// Only document what giselle-mui adds.
export interface CompositeCardProps extends PaperProps {
  /** Named slot for the card header region. Rendered above the body. */
  header?: ReactNode;
  /** Named slot for the card footer region. Rendered below the body. */
  footer?: ReactNode;
  /** Named slot for action buttons. Rendered at the bottom right. */
  actions?: ReactNode;
}
```

This is the productivity multiplier for large Sitecore teams: one import, full IDE documentation, no maintenance burden on the library author.

**4 — `...other` spread on root — non-negotiable for CMS compatibility**

The Sitecore Experience Editor attaches `data-*` attributes to every component for in-context editing. These must flow through to the DOM root element without configuration. The `...other` spread on the root element (already a rule in this library) is the mechanism. Every component that ships without it is broken in Sitecore editing mode.

**Prerequisite:** Phase E — `SectionContainer` and `HeroSection` must be reviewed for slot-API compliance before Phase J ships.

### Phase K — BranchedTimeline (Scenario Tree) — TBD ⚪ Good to have

> **Library classification open.** This component sits at the boundary of what a pure UI library should own. The branch layout algorithm is complex enough that it may warrant a dedicated package (e.g. `@alexrebula/giselle-smart-components`) rather than bundling into `giselle-mui`. The classification decision must happen before any implementation work begins.

**The problem it solves:**

`TimelineTwoColumn` and `TimelineCompact` are flat — they render a single ordered sequence of phases. When a roadmap or plan has branching futures (e.g. "if we ship in May → Scenario A plays out; if we delay to July → Scenario B plays out"), there is no way to represent both paths without duplicating data or collapsing branches into description text.

**What `BranchedTimeline` does:**

A timeline where the main spine can split into N parallel arms at a fork point. Each arm is its own mini-timeline with phases and milestones. Arms can re-converge to the main spine if all scenarios lead to the same outcome. Fork points carry a `condition` string ("Scenario A — Early launch") and a palette-keyed colour band that threads through the arm.

**Data model (proposed):**

```ts
type BranchedTimelineNode =
  | { type: 'phase'; data: TimelinePhase }                          // linear segment
  | { type: 'branch'; condition: string; arms: BranchedTimelineArm[] }; // fork point

interface BranchedTimelineArm {
  id: string;
  label: string;          // "Scenario A — Early launch"
  color?: PaletteKey;
  nodes: BranchedTimelineNode[];  // recursive — arms can contain further forks
}

interface BranchedTimelineData {
  title?: string;
  trunk: BranchedTimelineNode[];  // main spine before and after any fork points
}
```

This data shape maps directly to the `cleanup-workflow.md` document structure: each phase is a node, each scenario block is a fork, and the converging "Definition of done" section is a re-convergence node.

**Why the layout is the hard part:**

A flat timeline renders a single column of cards. A branched timeline must compute, per fork:

- How many parallel arm columns are needed
- How wide each column should be (proportional to arm complexity or fixed)
- Whether sibling-arm phases should align by date or by sequence index
- Whether arms re-converge (and where) — a re-convergence node must span all arm columns

This layout computation is a pure function with no React dependency — it belongs in a `utils.ts` and should be independently unit-tested before any JSX is written.

**Reuse of existing components:**

Phase cards and milestone badges inside each arm are existing `TimelineTwoColumn` internals (`PhaseCard`, `MilestoneBadge`, `SpineConnector`). The new work is the branch/re-convergence layout shell, not the per-item rendering.
